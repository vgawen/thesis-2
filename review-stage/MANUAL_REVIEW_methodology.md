## 一、总体判断

这是一篇**学术诚信度高、定位清醒的诊断型（diagnostic）论文**，而非"解决方案型"论文。

- 核心贡献是 **reframing**：揭示现有 LLM 定位器自愈研究普遍衡量"打过补丁的测试是否通过"，而非"补丁是否指向开发者真正期望的元素"，从而暴露**隐性误修复（silent false healing）**这一结构性盲区。
- 论文不解决该问题，而是为"必须引入 L4 行为重放验证器"提供**经验性动机**。这是合理的学术定位，但审稿人可能质疑贡献完成度。
- 最大优点：对抗性探针设计干净、F2 的 `Vanilla = Abstain`（Δ=0）反直觉但铁证、主动撤回早期虚高数字（98% → 80.6%）。
- 最大风险：**所有核心数字都是"静态解析层面"代理指标，由一个边界偏松的解析器单点裁决，且样本小、模型单一、主要单应用。**

**建议结论**：作为 short / diagnostic paper 可接受；若投强会议/期刊要求"完整方案 + 运行时验证"则贡献完成度会被质疑。

## 三、问题清单（按严重度排序，含 actionable 建议）

### [严重 / Major]
1. **解析器即 oracle 的单点依赖**。所有"正确匹配 / 误修复 / 可达率"判定由 `resolve_locators.py` 静态裁决，作者承认其"对动态 JS 表达式视为可能匹配、复合 CSS 用祖先过近似……边界偏松"。78.7% / 77.3% / 80.6% 全部悬挂于此。
   - **建议**：补一个 **strict / permissive 双解析器假设的敏感性扫描**，报告核心数字在两套假设下的区间。这是当前结论最脆弱的支点，优先级最高。

2. **静态代理与运行时通过率之间完全未校准**。全文无一个真正的 Playwright 运行时端到端数字；77.3% 仅为选择器解析层面精确一致。
   - **建议**：对抽样 10–20 个用例做**真实 DOM 快照 / Docker 重放交叉校验**，把"静态代理 ↔ 运行时真相"的鸿沟从"完全未知"降到"已抽样校准"。

### [中等 / Minor]
3. **15% 风险估计 → 强工程禁令的逻辑链偏单薄**。一个未实测的乘积（0.19 × 0.79）支撑"拒绝 CI 无人值守自动提交"。
   - **建议**：措辞进一步弱化为"风险量级提示"，或补一个真正混合工作负载（非对抗）下的小规模实测点。

4. **统计功效与外部效度**：单模型、主要单应用、F3 非系统采样。
   - **建议**：至少补一个第二模型（哪怕另一个开源 7B/14B）在 F1 上的点，以支撑"规模不足以闭合差距"这一推测；或将该推测进一步标注为未验证假设。

### [可选 / Nice-to-have]
5. F2 二因子析因消融（含 McNemar 配对检验）——作者已列为 future work，补上可显著加强 F2。
6. 78.7% 同时给出逐用例失败模式分布（错指到哪类元素），增强诊断说服力。
